# /translate-book

Запускает или продолжает полный пайплайн перевода книги `books/$1.pdf` на язык `$2` (по умолчанию `ru`).

Принцип работы — **идемпотентность через файловую систему**: всё состояние в `result/$1/translation/PROGRESS.md`, каждый запуск стартует с холодного чтения. Поэтому одна и та же команда работает и для первой инициализации, и для возобновления после паузы.

---

## Алгоритм

Выполняй пункты строго по порядку. После каждого шага, который меняет состояние на диске, **перечитывай `PROGRESS.md` заново** — не доверяй своей памяти.

### Шаг 0. Валидация входа

1. Прочитай `perevodnik/CLAUDE.md` (если ещё не прочитал в этой сессии).
2. Проверь `books/$1.pdf`. Если нет — остановись, сообщи пользователю, какие PDF лежат в `books/`.

### Шаг 1. Фаза INIT (только если `result/$1/` не существует)

1. Через `AskUserQuestion` собери у пользователя недостающие метаданные одним пакетом из 1–4 вопросов:
   - **Лицензия**: CC BY 4.0 / CC BY-NC-SA 4.0 / Public Domain / Other (пользователь впишет).
   - **Атрибуция оригинала** (строка): «OpenStax / Rice University», «Project Gutenberg», и т.п.
   - **URL оригинала** (опционально): ссылка на источник для секции «О книге».
   - **Repo URL** (опционально): где будет жить переведённый репозиторий (для кнопки «Редактировать» в MkDocs).
   `target_language=$2` уже известен, не спрашивай. Заголовок книги попробуй угадать из имени PDF и первых страниц `extracted/text.txt`, но это можно сделать ПОСЛЕ extract (см. шаг 3).
2. Создай `result/$1/book.yml`. Минимальный пример:
   ```yaml
   name: $1
   source_pdf: ../../books/$1.pdf
   title_original: "<заполнится после быстрого взгляда в PDF>"
   title_translated: "<временный заголовок — уточнится после картографа>"
   source_language: en
   target_language: $2
   license:
     type: <ответ пользователя>
     attribution: <ответ пользователя>
     attribution_url: <опционально>
   features:
     math: true   # отключить, если книга нетехническая
   parallelism:
     translators: 4
     reviewers: 2
   site:
     repo_url: <опционально>
   ```
3. Запусти: `python core/scripts/init_book.py result/$1/book.yml` — создаст скелет.
4. Запусти: `python core/scripts/extract.py books/$1.pdf result/$1/extracted/` — заполнит `text.txt` / `pages/` / `images/`.
5. Открой `result/$1/extracted/text.txt` (первые 50 строк) и пару `pages/page-0001.png` … `page-0005.png`. Уточни `title_original` и `title_translated` в `book.yml`.
6. Запусти агента-картографа (`Agent`, single, см. `core/templates/$2/AGENT_PROMPTS.md` роль A). Подставь имя книги. Результат — `result/$1/translation/chapter_map.json`.
7. Запусти: `python core/scripts/render_mkdocs.py result/$1/` — перегенерирует `mkdocs.yml` и `PROGRESS.md` из новой карты глав.
8. Сообщи пользователю: «Книга инициализирована. Начинаю перевод. Можно прерывать в любой момент — `продолжай` подхватит.»

### Шаг 2. Цикл работы

Повторяй пункты 2.1–2.3, пока есть незакрытая работа ИЛИ пока пользователь не остановит ИЛИ пока модель не упрётся в лимиты.

#### 2.1. Прочитать состояние

```
python core/scripts/progress.py result/$1/translation/PROGRESS.md
```

Получишь JSON с полями `pending`, `draft`, `review`, `ready_to_assemble`, `next_action`. Доверяй `next_action`.

#### 2.2. Выполнить следующее действие

- `next_action == "translate"`: возьми первые `book.yml: parallelism.translators` записей из `pending` и запусти переводчиков **одним сообщением с N вызовами `Agent`**. Промпт — из `core/templates/$2/AGENT_PROMPTS.md` роль B, с подставленными `chapter_map.json`-данными (строки/страницы/имя файла-результата).
- `next_action == "review"`: возьми из `draft` записи и запусти ревьюеров (роль C, тоже параллельно).
- `next_action == "assemble"`: возьми главы из `ready_to_assemble` (по одной за вызов) и запусти сборщиков (роль D, последовательно — они трогают общий `mkdocs.yml`).
- `next_action == "build_site"`: всё в `done`. Запусти `cd result/$1 && mkdocs build --strict`. Если зелёный — сообщи пользователю URL локальной сборки (`result/$1/site/index.html`) и спроси, нужно ли коммитить и/или открывать PR.
- `next_action == "idle"`: значит состояние странное (нет ни pending, ни draft, ни review, но и не build_site). Покажи пользователю текущий `PROGRESS.md` и спроси, что делать.

#### 2.3. Дождаться завершения батча и вернуться к 2.1

**Не** иди дальше по своей памяти о том, какие разделы остались. Каждая итерация — холодное чтение `PROGRESS.md`. Это и есть механизм возобновления.

### Шаг 3. Завершение

Когда `next_action == "build_site"` и сборка успешна:

1. Сообщи пользователю краткую сводку: «Переведено X разделов в Y главах. Сайт собран в `result/$1/site/`. Открыть локально: `cd result/$1 && python -m http.server -d site 8000`».
2. Спроси (текстом, не AskUserQuestion), нужно ли:
   - закоммитить изменения,
   - запушить,
   - открыть PR,
   - запустить `mkdocs serve` для предпросмотра.
3. Без явной просьбы пользователя — ничего из этого не делай.

## Параллельный запуск агентов

Когда тебе нужно запустить N переводчиков или N ревьюеров — отправь **одно сообщение с N тегами `Agent`** (не N последовательных сообщений). Это критично для скорости.

Не превышай `book.yml: parallelism.translators` для переводчиков и `parallelism.reviewers` для ревьюеров.

## Безопасность

- Никогда не редактируй `books/$1.pdf`, `result/$1/extracted/`, `result/$1/site/`.
- Никогда не вызывай `mkdocs gh-deploy`, не делай `git push` без явной команды пользователя.
- Если упёрся в лимит модели или агенты возвращают ошибки — остановись и сообщи пользователю. PROGRESS.md уже отражает реальное состояние, продолжить можно будет в следующий запуск.
