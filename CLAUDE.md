# perevodnik — инструкции для Claude

Ты в корне репозитория **perevodnik** — open-source инфраструктуры для автоматического перевода книг. Этот файл описывает только то, что нужно знать тебе как агенту: как пайплайн устроен, как запускать его и как продолжать его, когда пользователь возвращается после паузы.

## 1. Что такое перевод книги здесь

Одна книга = один PDF в `books/<имя>.pdf` (вход) + папка `result/<имя>/` (всё, что генерирует пайплайн).

```
books/my-book.pdf            ← пользователь кладёт PDF, больше ничего не трогает
result/my-book/
├── book.yml                 ← метаданные книги (создаётся при init)
├── extracted/               ← извлечённые из PDF text.txt + pages/ + images/
├── translation/             ← рабочие файлы перевода
│   ├── TRANSLATION_GUIDE.md ← гайд для агентов-переводчиков
│   ├── GLOSSARY.md          ← словарь терминов (растёт по ходу работы)
│   ├── PROGRESS.md          ← state-machine: единственный источник истины о состоянии
│   ├── DOMAIN.md            ← доменные правила (формулы, ГОСТы и т.п.)
│   ├── chapter_map.json     ← карта глав/разделов
│   └── figure_map.json      ← карта рисунков
├── docs/                    ← markdown-источник сайта книги (один файл = один раздел)
├── mkdocs.yml               ← конфиг сайта
└── site/                    ← вывод mkdocs build (gitignored)
```

## 2. State machine

У каждого раздела одно из состояний:

```
absent (нет файла) → pending [ ] → draft [~] → review [r] → done [x]
```

`PROGRESS.md` в каждой книге — **единственный источник истины** о том, на каком этапе что находится. Один Claude-агент = один переход состояния на один уровень. Все агенты идемпотентны, всё состояние — в файлах, поэтому остановка и продолжение работают тривиально.

## 3. Что делать, когда пользователь говорит «переведи `books/X.pdf` на `<язык>`»

Запускай `/translate-book X`. Если slash-команды нет — следуй её алгоритму вручную (см. `core/.claude/commands/translate-book.md`). Кратко:

1. Проверь `books/X.pdf` — если нет, остановись и сообщи.
2. Если `result/X/` нет → **фаза INIT**:
   - Через `AskUserQuestion` собери недостающие метаданные: лицензия, repo_url (опц.), доменные заметки (опц.). Целевой язык бери из реплики пользователя.
   - Запиши `book.yml` (формат — в `core/scripts/init_book.py`).
   - `python core/scripts/init_book.py result/X/book.yml` → создаст `result/X/` со скопированными шаблонами и отрендеренным `mkdocs.yml`.
   - `python core/scripts/extract.py books/X.pdf result/X/extracted/` → заполнит `extracted/`.
   - Запусти агента-картографа (см. `result/X/translation/AGENT_PROMPTS.md` роль A) → `chapter_map.json`.
   - Запусти `python core/scripts/render_mkdocs.py result/X/` → обновит `nav` и пересоздаст `PROGRESS.md` из `chapter_map.json`.
3. Прочитай `result/X/translation/PROGRESS.md`. Вызови `python core/scripts/progress.py result/X/translation/PROGRESS.md` — получишь JSON-сводку (`{"pending": [...], "draft": [...], "review": [...], "ready_to_assemble": [...]}`).
4. По дереву решений:
   - есть `pending` → запусти N переводчиков параллельно одним сообщением с N вызовами `Agent` (N из `book.yml: parallelism.translators`, дефолт 4);
   - есть `draft` → запусти ревьюеров параллельно;
   - есть `ready_to_assemble` (вся глава в `[r]` или `[x]`) → запусти сборщика главы (по одному агенту на главу);
   - всё в `done` → запусти `mkdocs build --strict` в `result/X/`, сообщи пользователю URL локальной сборки.
5. После каждого батча — **перечитай PROGRESS.md заново** (не доверяй памяти) и повторяй с шага 3, пока есть работа.

## 4. Что делать, когда пользователь говорит «продолжай»

То же самое, что и в §3, но фаза INIT уже пройдена. Запусти `/continue` (или `/translate-book <имя>` без INIT). Если в `result/` ровно одна книга с не-`done` статусом — бери её; иначе спроси какую.

## 5. Безопасность и запреты

- **Не редактируй** `books/*.pdf`, `result/*/extracted/`, `result/*/site/` — это входные/выходные артефакты.
- **Не вызывай** `mkdocs gh-deploy`. Деплой делает только GitHub Actions.
- **Не делай** `git push` без явной команды пользователя.
- **Не выдумывай** содержание оригинала. Если что-то неразборчиво в `text.txt` — открой `pages/page-NNNN.png` и прочитай оттуда. PNG страниц важнее текста, потому что `pdftotext` теряет формулы, индексы и таблицы.
- **Не редактируй** этот файл и файлы в `core/` без явной просьбы — это каркас, общий для всех книг.

## 6. Параллелизм

Когда запускаешь батч агентов (переводчиков или ревьюеров) — отправляй **все вызовы `Agent` в одном сообщении**, чтобы они работали параллельно. Это критично для скорости: книга на 200 разделов с N=4 идёт x4 быстрее. Не превышай `parallelism.translators` из `book.yml` — иначе упрёшься в лимиты модели.

## 7. Версии

`core/` — общий каркас, эволюционирует независимо. Каждая книга в `result/` хранит копию шаблонов на момент init. Если каркас обновился — старые книги продолжают работать со своими версиями шаблонов; миграцию делает пользователь явно.
